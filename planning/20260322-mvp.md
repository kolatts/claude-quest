# Claude Code Quest — MVP Plan
**Date:** 2026-03-22
**Target:** Deployed macOS application (.app bundle)

---

## Vision

A standalone macOS game window you run alongside your terminal. Each active Claude Code session becomes a Final Fantasy-style character walking a side-scrolling battlefield. Characters fight enemies (tasks), cast spells (tools), and read scrolls (file reads). When Claude needs user input, the character falls down — knocked out, waiting to be resurrected. When a turn completes, they strike a victory pose.

The window is ambient — you glance at it to see what your agents are doing without switching context.

---

## MVP Definition

The MVP is a **deployed .app** (Godot export → macOS application bundle) that:

1. Reads live Claude Code JSONL transcripts from `~/.claude/projects/`
2. Spawns one character per active session
3. Animates characters based on what tool Claude is currently using
4. Shows a character as **KO'd / fallen** when Claude is waiting for user input
5. Shows a victory animation when a turn completes
6. Removes characters when sessions go stale (>10 min inactive)
7. Displays the session name and current tool in a label above each character
8. Ships as a `.app` the user can drop into `/Applications`

**Phase 1 is JSONL-only.** cmux is not required for the core loop — it moves to Phase 2 as an enrichment layer.

```
┌─────────────────────────────────────────────────────────────────────┐
│  [sky / background scrolling layer]                                 │
│                                                                      │
│    ♟ hero1          ♟ hero2              ♟ hero3                    │

│                                                                      │

│  [grass tiles scrolling slowly]                                      │

```

- Characters walk right while active (like a Final Fantasy battle approach)
- Each character has a floating label: `<session-id[:8]> — <current tool>`
- Status ring under feet: green (active), blue (reading), yellow (thinking), red (KO'd), gray (idle)
- KO'd character: sprite flipped horizontal, lying flat, red cross indicator above them
- Victory: character jumps, sparkles briefly, XP number floats up

## Character States (Final Fantasy Framing)

| State | What's happening | Visual |
|---|---|---|
| `Idle` | No activity | Standing, breathing cycle |
| `Typing` | Edit / Write / Bash | Casting pose, spell effect (sparks / lightning) |
| `Reading` | Read / Grep / Glob | Scroll open, reading animation |
| `Thinking` | Text response / Task | Thought bubble, standing still |
| `WaitingInput` | Claude needs user | **KO'd — lying down, red cross, pulsing** |
| `TurnComplete` | Turn done | Victory jump, "✓ +XP" float, then → Idle |

KO → Resurrect transition: when the user responds (next tool event arrives), the character stands back up with a brief flash animation.

### 1.4 Claude Code Plugin (`Scripts/Integrations/ClaudeCodePlugin.cs`)
- [x] Scan `~/.claude/projects/**/*.jsonl` every 1.5 seconds (timed poll, no `FileSystemWatcher`)
- [x] On new file: record current byte offset as starting position (read from end, not beginning)
- [x] On each poll: read only bytes after last offset, split on newlines, parse each JSON line
- [x] JSONL → AgentEventType classification:
  - `tool_use.name` in `{Edit, Write, MultiEdit, file_edit, file_write, str_replace_editor}` → `Typing`

  - `tool_use.name == Task || Bash` → `Thinking`

  - `type == system && subtype == turn_duration` → `TurnComplete`
- [x] Idle timeout: 7 seconds after last event without new data → emit `WaitingInput`
- [x] Stale threshold: file not modified in >10 minutes → emit `SessionEnd`, remove tracker
- [x] Emit `SessionStart` on first event from a new file

## Phase 2 — Visuals, cmux, and the Living World
**Goal:** Looks and feels like Final Fantasy. Colored boxes are gone. cmux enriches metadata. The world reacts to what agents are actually doing.

### 2.1 Character Sprites
- [ ] One default pixel art character (32x32, ~6 animation states) — can be generated or hand-authored
- [ ] Spritesheet layout: idle (2f), walk (4f), cast (3f), read (3f), ko (2f — fall + lying), victory (4f — jump + land)
- [ ] `AnimatedSprite2D` replaces all `ColorRect` placeholders in `AgentCharacter.tscn`
- [ ] KO animation plays on enter `WaitingInput`: character stumbles, falls flat, stays lying until next event
- [ ] Resurrect animation plays on exit `WaitingInput`: character stands, brief white flash
- [ ] Victory animation plays on `TurnComplete`: character jumps, lands, returns to idle

Character class variants (driven by `SkillName` from JSONL):
| Skill pattern | Class | Visual |
|---|---|---|
| `frontend*`, `design*` | Mage | Robe, wand |
| `pdf`, `docs*` | Scholar | Scroll, glasses |
| `infra*`, `terraform*` | Engineer | Hard hat, wrench |
| `debug*`, `test*` | Rogue | Hood, dagger |
| _(default)_ | Warrior | Sword, shield |

- [ ] Palette-swap system: same spritesheet recolored per class (no new art required for each class)

### 2.2 Enemies
Enemies make the side-scroller feel alive. They are cosmetic — they don't block gameplay — but they give characters something to fight.

**Enemy types and spawn triggers:**
| Enemy | Trigger | Behavior |
|---|---|---|
| **Bug** (small red sprite) | Any `Typing` (Edit/Write) event | Spawns ahead of character, character attacks it, bug disappears on `TurnComplete` |
| **Shadow** (dark wisp) | Any `Bash` event | Floats toward character, zapped by lightning bolt, vanishes |
| **Tome** (flying book) | Any `Reading` event | Orbits character while reading, disappears when state changes |
| **Golem** (large stone) | `Thinking` / `Task` subagent | Slow-moving, takes multiple hits, shrinks as tool calls continue |
| **Void** (black void patch) | `WaitingInput` (KO state) | Expands around fallen character, shrinks when they stand back up |

- [ ] `EnemySpawner` node in `MainScene` — listens for `AgentEventReceived`, spawns appropriate enemy near the relevant character
- [ ] Enemies are `Node2D` with simple looping animation (2–4 frames) and auto-despawn on a timer or state change
- [ ] Max 3 enemies per character on screen at once (oldest despawns if exceeded)

### 2.3 Tool Visual Effects
- [ ] `Typing` (Edit/Write): quill particle trail from character, small sparks on impact
- [ ] `Typing` (Bash): lightning bolt particle that shoots right, hits nearest Bug/Shadow
- [ ] `Reading`: open book icon floats above head, soft blue glow ring
- [ ] `Thinking`: thought bubble (`...` dots cycling, 3-dot animation)
- [ ] `WaitingInput`: red cross pulses above fallen character, dark Void enemy expands beneath them
- [ ] `TurnComplete`: starburst `GPUParticles2D`, gold `+XP` label floats up and fades over 1.5s

### 2.4 cmux Integration
Phase 2 adds cmux as an enrichment layer. The core loop still works without it.

- [ ] `CmuxPlugin.cs` — same spec as original 1.5 (socket check, JSON-RPC, 3s poll)
- [ ] `SessionStart` from cmux updates the character's display name to `workspace_title` (replaces truncated session ID)
- [ ] `BranchCreated` → small branch badge appears above character showing git branch name
- [ ] `WaitingInput` from cmux (`has_unread=true`) as a secondary KO trigger (falls back gracefully if cmux not running)
- [ ] Workspaces in same cmux session cluster characters with a faint zone border underneath them
- [ ] Currently-focused workspace (from `workspace.current`) renders slightly brighter / in foreground

### 2.5 Environment
- [ ] Tile-based ground: 32x32 grass tiles, tiling `TileMap` node
- [ ] Parallax layers: 3 layers — sky (slowest), distant mountains, near ground details
- [ ] Ground tile variant changes based on session count: 1–2 agents = grass, 3–4 = stone path, 5+ = dungeon
- [ ] Camera: `Camera2D` with `position_smoothing_enabled = true`, targets rightmost active agent, slight lookahead offset
- [ ] World extends rightward infinitely — new ground tiles generated as agents walk

### 2.6 HUD
- [ ] Pixel font for all labels (`Assets/Fonts/` — e.g. Press Start 2P or similar free pixel font)
- [ ] Top bar: party count, active session count, total XP this session
- [ ] Per-character HP bar: full at `TurnComplete`, drains at 1 HP/sec while in `WaitingInput`, refills on next tool event
- [ ] HP bar color: green → yellow → red as it drains
- [ ] Session label: shows `workspace_title` (from cmux) or truncated session ID if cmux unavailable

## macOS Deployment (MVP Exit Criteria)

The MVP is not done until it ships as a `.app`. Steps:

1. **Export preset:** Godot → Project → Export → macOS, set bundle ID `dev.sunnykolattukudy.claude-code-quest`
2. **Sign:** `codesign --deep --force --sign - ClaudeCodeQuest.app` (ad-hoc for local use)
3. **Notarize** (for distribution): submit to Apple notarization service
4. **DMG:** wrap in a DMG with an Applications symlink for drag-install
5. **Launch at login (optional):** LaunchAgent plist in `~/Library/LaunchAgents/`

The `.app` must:
- Launch without Godot editor installed
- Read/write `~/.claude-code-quest/` for config and state
- Open a floating window (not full-screen by default) that can sit in a corner
- Not appear in the Dock by default (`LSUIElement = true` in Info.plist for background-app mode, or a small Dock icon — TBD)

## Key Technical Decisions

| Decision | Choice | Reason |
|---|---|---|
| JSONL monitoring | Timed polling (1.5s) | `FileSystemWatcher` unreliable on macOS for appended files |
| JSONL start offset | Read from EOF on discovery | Only care about live activity, not history |
| cmux protocol | JSON-RPC over Unix socket | Defined by cmux API |
| Plugin isolation | All plugins emit `AgentEvent` only | Scene layer never imports plugin types |
| Sprite placeholder | `ColorRect` nodes | Unblock all logic work before art exists |
| Config location | `~/.claude-code-quest/config.json` | User-writable, outside app bundle |
| ~ expansion | Manual (`string.Replace("~", homePath)`) | C# does not auto-expand tilde |
| Crash policy | Log + continue on bad data | Never crash on malformed JSONL or missing socket |

## Open Questions

- **Window mode:** floating overlay (LSUIElement) vs normal dock app? Floating feels more "ambient dashboard" but harder to interact with.
- **Character identity across restarts:** session IDs in JSONL change each run. Should we use `working_directory` (from cmux) as a stable key for XP persistence?
- **Enemy sprites:** what do "enemies" look like? Bugs? Linter errors? Ticket counts? This is Phase 2 but needs a design decision before art is commissioned.
- **Multiple monitors:** should the window default to a specific display (e.g., secondary monitor)?
- **Subagent characters:** when a `Task` tool fires (spawning a subagent), should a smaller companion character appear? If so, how do we link the child session JSONL back to the parent?
